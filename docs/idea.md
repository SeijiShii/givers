# 「与える」プラットフォーム GIVErS

## 第一目標

- **日本で寄付の文化を根付かせることを第一目標とする。**
- そのため、**日本での寄付に必要な情報の提供を最優先**する。寄付金控除（税額控除）・領収書の扱い、日本語での案内・FAQ など、日本の寄付者が安心して寄付するために必要な情報を優先的に整える。他言語・他地域向けの情報は、日本での情報提供を足がかりに拡張する。

## 思想

### GIVE の精神

- シェアウェアの考え方に近い
  - アプリやサービスを無料で開発、提供する
- ボランティアだと持続性が難しい
- クラウドファンディングだと目標設定やリワード設定が必要
- Gitcoinはまだまし。
- 自分が「世界の役に立つ」「絶対これは良い」と思うものを作りたい。
  - マネタイズしようとすると思想が汚損する。
  - やりたいことではなくマーケットが求めるものになる
- それを使った人が「これは素晴らしい。応援したい」と考える
- 両方に大切なのは GIVE の精神
  - **作り手の GIVE**: 見返りを求めず、自分が良いと思うものを作る
  - **受け手の GIVE**: 使って良かった人が、自発的に応援したいと思う

### プラットフォーム自体の GIVE

- **GIVErS は手数料を取らない**
- GIVErS 自体が GIVE で運営される
- まわらなくなったら、世に受け入れられなかったということ
- 原理を曲げてまで存続しない。存続は「世の中の評価」に委ねる
- ある日突然クローズする可能性もあるため、利用者に申し訳ない思いを残さないよう透明性を保つ

### サービスホストについて

- 「サービスホストについて」メニューからホストページを表示できる
- ホストページでは、GIVErS が健全に運営されているかどうかを閲覧できる
- バナーに青信号・黄信号・赤信号のようなマークで健全性を表示する
  - 🟢 健全（余裕あり）
  - 🟡 注意（ギリギリ）
  - 🔴 危険（継続困難）
- **ホストはアカウントコントロール権限を持つ**。ユーザー一覧にアクセスでき、プロジェクトオーナーがシステムを悪用していると判断する場合、利用停止できる。停止解除もできる。**利用停止したユーザーに紐づく定期寄付（Stripe サブスク）は、停止と同時に解約する**。悪用者から継続入金させないため。詳細は `docs/mock-implementation-status.md` 12.5 参照。
- **利用停止されたアカウント**でログインしているユーザーが寄付・プロジェクト作成等のアクションをしようとしたときは、**状況を理解できる親切なメッセージ**を表示する（例：「このアカウントは利用停止されています。ご不明な点は運営までお問い合わせください。」）。単に 403 を返すだけでなく、画面で理由が分かるようにする。

### 人の善性を信頼する

- GIVErS を騙って資金集めする輩も出る想定
- 思想としては「相手にしない」
- 対策に振り回されず、人の善性を信頼する
- 悪い少数のために、良い多数に負担をかけない

## 現実的であること

- これを実現するにはいつまでどれくらいの資金が必要か
- このサービスを継続するには月々どれくらいの費用が必要か
- Wikipediaなどがやっているように「サービスの継続にはあなたの寄付が必要です」という広告を出すよりも、定量化された情報を表示できる
- 定量化すると「このサービスは継続できるのだな」「継続性がないのにがんばっているのだな」見えてくる
- 寄付者視点からは、自分がプロジェクトの何％くらいを支えているのかが見えてくる。

## 募集者マイページ（プロジェクトの費用設定）

プロジェクトオーナーはプロジェクトを**立ち上げ**、**凍結**、**削除**できる。DB 的には論理ステータス管理（active / frozen / deleted など）で、物理削除は行わない。**凍結時**は新規寄付・新規サブスクの受付を停止し、既存サブスクは継続。**削除時**は新規受付停止に加え、既存サブスクも解約する。詳細は `docs/mock-implementation-status.md` 12.5 参照。
- **凍結・削除されたプロジェクト**に寄付やその他のアクションをしようとしたユーザーには、**状況を理解できる親切なメッセージ**を表示する（例：「このプロジェクトは現在、寄付の受付を停止しています。」「このプロジェクトは終了しています。」）。API が 403 や 404 を返す場合でも、フロントで理由が分かる文言を表示する。

募集者・プロジェクトオーナーのマイページで、以下をブレークダウンして設定できる。

| 項目 | 設定内容 |
|------|----------|
| プロジェクトの期限 | いつまでか、永続か |
| サーバー費用 | 月額 |
| 開発者の人日コスト | 1日いくら、何日稼働するか |
| その他の費用 | 月額 |
| **オーナーの最低希望額** | 「最低〇〇円欲しい」と表明する月額（任意）。必要額への上乗せではなく、作り手が受け取りたい最低ライン。 |

これにより必要な月額コストが定量的に計算され、プロジェクトページに公開される。オーナー希望額は、コスト内訳とは別に「作り手の率直な要望」として表示される。寄付者には「このプロジェクトは月額〇〇円必要」「現在〇％達成」などが透明に表示される。

目標金額に加えて、実際の稼働コストや経費を公開できる。計画値だけでなく実績値も見せることで、現実味が増し、寄付者の信頼につながる。

### オーナーの「欲しい」表明（あっけらかんとした要望）

コストの内訳とは別に、**作り手本人が「最低〇〇円欲しい」とあっけらかんと表明できる**仕組みを用意する。

- **必要額**: 「サーバー〇円、開発〇円…」という定量的なコスト内訳
- **最低希望額**: 「最低〇〇円欲しい」という作り手の受け取りたい最低ライン（必要額への上乗せではない）

両方を表示できる。希望額は必要額に足し算する金額ではなく、作り手が「これ以下では続けられない」と表明する最低ライン。コストより低く設定することも（自腹で補填する意思）、高く設定することも（人件費などを含めたい）できる。透明性と人間味の両立を目指す。

### プロジェクト単位の危険度レベル（アラート）

- プロジェクトごとに WARNING、CRITICAL などの危険度閾値を設定できる
- 例: 月額達成率が 50% を下回ったら WARNING、20% を下回ったら CRITICAL
- 実際の数値が閾値を割るとアラート状態になり、プロジェクトページに表示される
- 寄付者に「このプロジェクトは今、支援が足りていません」と伝えやすくなる
- **赤信号は「やめなくてはならない」という意味ではない**。コストの低いプロジェクトなら、気持ちで続けられる限り赤信号のままでも続けてよい
- ただし、気持ちはあっても世界に支持されていないという現実を示す。透明性のための表示である

## 参加方法・リンク

- 利用者は GIVErS ページへのリンクを GitHub リポジトリなどに配置できる
- その他のプロジェクトのページからリンクできる
- 公式サービスにアカウントを作ってもいいし、OSS でクローンしたものを自分のサーバーにホストしても良い
- どちらも「正しい参加方法」。プラットフォームへの依存を強制しない

## サイト構成・ナビゲーション

- 寄付者は募集者のサイトなどから募集者のプロジェクトページに飛んでくる
- プロジェクトページから他のプロジェクトページにも遷移できる。発見のきっかけを提供する
- トップページには新しいプロジェクトや HOT なプロジェクトを、何らかのアルゴリズムで表示する

## 消極的防衛（なりすまし対策）

思想を壊さない範囲で、利用者が「公式かどうか」を判断できる情報を用意する。**自ホストには寛大であり、公式かどうかにはあまりこだわらない**。フォークしてリンクを削除するならそれもあり、とする。

### やること

| 施策 | 内容 |
|------|------|
| About での案内 | **About ページに公式の URL と GitHub リンクを載せておく**。利用者が参照できる程度にとどめる。フッターや初期表示での強めの「公式」表記は必須としない。 |
| 自ホスト | 自ホスト版での「公式」表記の削除・変更は自由。自ホスト版の自己申告（「このサイトは GIVErS の自ホスト版です」等）は任意で、実装するなら環境変数で出し分け可能にする程度。 |

### やらないこと

| やらないこと | 理由 |
|-------------|------|
| 認証バッジ・マークの発行 | 能動的になり、疑いの文化を作る |
| 偽サイトのリスト維持 | 監視・管理コストが増える |
| 通報窓口 | 対応負荷と能動的対応になる |
| 複雑な検証フロー | 善意前提の UX を壊す |

### 公式のスタンス（責任について）

- **悪用者が悪意を持ってクローンサービスを運用する場合、公式（GIVErS の運営）は責任を負わない**ことを公言する。
- OSS として配布・フォークを許容する一方、第三者による悪用については関知しない旨を明示する。About や利用規約・README で記載することを検討する。

## 決済方法

- Stripe決済
- 様々な通貨、口数単位、定期購入などで柔軟な支援ができるようにする。
- **GIVErS プラットフォームは寄付金の授受を仲立ちしない。** 寄付金は寄付者から、Stripe Connect を通じて各プロジェクトのオーナー（募集者）に直接入金される。プラットフォームは入金の経由地とならない。

### 募集者の設定オプション

募集者（プロジェクトオーナー）が GIVErS 上で以下を設定できる。Stripe の仕様上、いずれも実現可能。

| 項目 | 設定内容 |
|------|----------|
| 金額 | 推奨額（¥500, ¥1,000 など）の設定、任意額の許可 |
| 通貨 | 受け取り通貨（JPY, USD, EUR など）を複数選択可 |
| 単発 / サブスク | 単発のみ / サブスクのみ / 両方 を選択 |
| サブスク周期 | 月額 / 年額 など |

技術的には Stripe Connect で各プロジェクトに直接入金し、募集者の設定に応じて Price / Product を動的に作成する形が想定される。

**寄付金控除（税額控除等）**: 寄付者が税額控除等の恩恵を受けるには、受け取り側が認定NPO法人等であることと、領収書（金額・日付・氏名・住所）の発行が必要。領収書の発行主体は受け取ったオーナー（団体）であり、プラットフォームは発行しない。**税額控除の対象となる NPO 法人（認定NPO法人等）の場合のみ**、その種の書類（領収書等）の発行を**手助けする**仕組み（「税額控除対象」の表明表示・寄付者氏名・住所の取得・オーナー向け寄付一覧のエクスポート等）を提供する。詳細は `docs/tax-deduction-considerations.md` を参照。

## 寄付者 UX（トークン → アカウント移行）

- **システムは、アカウントの有無を問わず、全ての寄付者の寄付行動履歴を保存する。**（単発・匿名寄付もトークンに紐づけて記録し、アカウント作成後に引き継ぎ可能にする。将来的にはデータ分析などに活用できるようにする。）
- 散発的支援者は最初トークン（Cookie）でトレースする。アカウント不要。
- 継続的支援に移行したくなったらアカウントを作る（Google ログインなど）。
- アカウント作成時、同じブラウザにトークンがあれば「これまでの寄付履歴をアカウントに引き継ぎますか？」と案内し、トークンに紐づく寄付をアカウントに移行する。
- **匿名寄付後の案内**: 寄付完了画面（サンクス）で、未ログインで寄付した場合に「アカウントを作成すると、このブラウザでの寄付履歴を引き継げます。」と 1 文で案内する。技術用語（Cookie）は出さない。
- 継続支援はアカウント必須（解約・変更を確実に管理できるようにするため）。
- **1アカウントでプロジェクトホストも寄付もできる**。プロジェクトをホストする人と寄付する人の区別はなく、同じアカウントで両方の役割を担える。
- **プロジェクトオーナーは複数のプロジェクトを立ち上げて募集できる**。
- **同一ユーザー判定**: Google・GitHub・Apple・Email など複数のログイン手段で、**同じ email であれば 1 アカウントにまとめる**（email をキーにアカウントをリンクする）。別プロバイダで同じメールアドレスでログインした場合は既存ユーザーに紐づけ、寄付履歴・プロジェクトは 1 つに統合する。
- **データ・プライバシー**: 寄付履歴は永続保存を原則とする。データ分析は匿名集計およびプロジェクトオーナー向け自プロジェクト集計に限る。**アカウント削除依頼**には応じ、その場合寄付レコードの donor は匿名化して行は残す（集計整合性のため）。利用停止時は寄付履歴は変更しない。詳細は `docs/mock-implementation-status.md` 12.3 参照。

## プロジェクトオーナーのミュート機能

- **システム仕様**: メッセージは**寄付時のみ**送れる。寄付フォームのメッセージ欄が唯一の送信手段。
- ただし、寄付をしながら悪態をつく器用な人がいるかもしれない。悪意のあるユーザーへの対策として、**プロジェクトオーナーは他のユーザーをミュートできる**。
- 悪意のあるユーザーを**ミュート**したり、**解除**したりできる。
- **ミュートされている場合、そのプロジェクトに寄付できない**。
- **ミュートはプロジェクト単位で管理**する。プロジェクトAでミュートされても、プロジェクトBには寄付できる。

## デプロイ方法

- サービスプラットフォームとしてホストする。
- OSSとして公開し、誰でもホストできるようにする。
- このプロジェクト自体を GIVErS プラットフォームで資金調達する。
- GIVErS を支援するのは主にプロジェクトオーナーたち。プラットフォームの恩恵を受ける人が、プラットフォーム自体も支える構造になる。
