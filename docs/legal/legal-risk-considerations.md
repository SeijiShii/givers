# 運営上の法的リスクの検討メモ

検討・仮実装した仕様のうち、**運営者が悪用者の悪い振る舞いにより法的に不利になる可能性**や、**詐欺の片棒を担いだ等の提訴を受けうる部分**を整理する。法務助言ではなく、弁護士確認・検討のきっかけとするためのメモである。

**弁護士に送付する際**: 相談内容を一括して把握しやすくした文書として **`docs/legal-consultation-brief.md`** を用意している。同書に「このサービスがやりたいこと」「想定している法的リスク」「どのような対応を考えているか」「実装済みの対策」「弁護士に相談したい論点」をまとめてある。

**最終更新**: 2025-02

---

## 0. 実装済みの対策（リスク低減・説明責任）

以下の対策は**すでに仕様・UI・文書に反映済み**である。

| 対策 | 内容 |
|------|------|
| **寄付金を仲立ちしないことの明示** | 寄付金は Stripe Connect でオーナーに直接入金することを FAQ・About・仕様書で明示。資金の経由地にならないことを明確化。 |
| **利用停止時のサブスク解約** | 利用停止したユーザーに紐づく定期寄付（Stripe サブスク）は、停止と同時に解約する方針を文書化・実装。悪用者から継続入金させない。 |
| **利用停止・凍結時のメッセージ** | 利用停止アカウントで寄付・作成等を試みた場合、「このアカウントは利用停止されています。ご不明な点は運営までお問い合わせください。」等を画面表示。凍結・削除プロジェクトに寄付等を試みた場合も「寄付の受付を停止しています。」「プロジェクトは終了しています。」等を表示。403 だけでなく理由が分かるようにしている。 |
| **クローン・悪用について「責任を負わない」公言** | About ページに「悪意あるクローン運営について公式は責任を負わない」旨、および「開示の要否・範囲は弁護士等の助言に従う」旨を記載済み。 |
| **公式 URL・GitHub の明示** | About に公式サイトの URL と GitHub リポジトリへのリンクを記載。利用者が「公式かどうか」を参照できるようにしている。 |
| **開示用データの出力機能** | 第三者情報開示請求等に備え、管理画面からユーザーID または プロジェクトID を指定して開示用データを JSON で出力する機能を実装。開示の要否・範囲は弁護士等の助言に従う前提。 |

## 1. リスクが論点になりうる領域

### 1.1 公式サイト上での詐欺的プロジェクト（寄付の詐欺幇助）

| 仕様・現状 | リスクの論点 |
|------------|--------------|
| 寄付金はプラットフォームが仲立ちせず、Stripe Connect でオーナーに直接入金（idea.md）。 | それでも**寄付の機会・決済導線を提供している**ため、虚偽のプロジェクトで寄付を集めた悪意あるオーナーがいた場合、「プラットフォームが詐欺を助長した」と指摘される余地がある。 |
| プロジェクトの**事前審査・本人確認（KYC）**の明文化がなく、ホストの「悪用と判断した場合の利用停止」は**事後対応**。 | 事前の審査・表明の検証が弱いと、「不十分な対策で被害を拡大した」と主張されるリスク。 |

**検討の提案**: 利用規約で「プロジェクト内容・オーナーの表明についてプラットフォームは保証しない」「寄付の判断は寄付者自身の責任」等を明記する。プロジェクト掲載時のガイドライン・禁止事項を定め、違反時の凍結・削除と併せて運用方針として残す。必要に応じて**弁護士**に、仲介役の範囲と免責の書き方の確認を依頼する。

---

### 1.2 「責任を負わない」公言の効力

| 仕様・現状 | リスクの論点 |
|------------|--------------|
| 悪用者が悪意を持ってクローンサービスを運用する場合、公式は責任を負わないと公言（idea.md）。 | **免責の明示**はリスク低減に役立つが、**法律上どこまで有効かは法域・事案による**。消費者契約法（免責条項の無効）、不法行為の故意・重過失による免責の制限など、無効・制限される場合がある。 |
| OSS 配布・フォークを許容し、第三者悪用については「関知しない」旨を明示。 | 裁判所が「合理的な注意義務（README・About での明示等）を果たしたか」を判断する材料にはなるが、**完全な免責は保証されない**。 |

**検討の提案**: About・利用規約・README に記載する文言は**弁護士の確認**を推奨。クローン運営と、**公式サイト上で発生する詐欺・紛争**は別論として、公式運営についての免責・責任制限も利用規約で整理する。

---

### 1.3 税額控除・領収書まわり（誤表示・誤認助長）

| 仕様・現状 | リスクの論点 |
|------------|--------------|
| プラットフォームは書類を発行しない。税額控除対象 NPO の場合のみ、オーナーの表明表示・氏名・住所取得等の「手助け」を検討（tax-deduction-considerations.md）。 | **認定NPO等でないオーナー**が「税額控除対象」と偽って表明した場合、寄付者が誤認し、プラットフォームの表示・仕様がその誤認を助長したと指摘される可能性。 |
| 表明はオーナーが行い、根拠（認定番号等）を登録する想定。 | プラットフォームがその表明を**無審査で表示**する場合、「虚偽表示を許した」と争われる余地がある。 |

**検討の提案**: 「税額控除の可否は寄付者・オーナーが各自で確認すること」「プラットフォームは表明の真正性を保証しない」旨を表示・利用規約に明記。認定番号等の**簡易照合**（公的一覧との突き合わせ）を検討する場合は、範囲と責任の切り分けを弁護士に確認する。

---

### 1.4 利用停止・凍結の判断（恣意性と不作為）

| 仕様・現状 | リスクの論点 |
|------------|--------------|
| ホストが「プロジェクトオーナーがシステムを悪用していると判断した場合」利用停止できる（idea.md）。 | 判断基準が**曖昧**だと、(1) **不当な利用停止**として訴えられる、(2) 逆に**明らかな悪用を見過ごした**として「不十分な対策」で訴えられる、の両方が論点になりうる。 |

**検討の提案**: 利用規約に「悪用」の定義・例（禁止行為）と、停止・凍結の手続きを簡潔に記載する。ホストの裁量を残しつつ、「合理的な根拠に基づく対応」が説明できるようにする。

---

### 1.5 本人確認・マネロン対策

| 仕様・現状 | リスクの論点 |
|------------|--------------|
| 認証は OAuth（Google/GitHub/Apple）・Email マジックリンク。**実名確認・本人確認（KYC）**の記載がない。 | 寄付の受領者が法人・団体の場合、**マネロン規制**や**犯罪収益移転防止法**等の議論に触れる可能性がある。日本では「寄付プラットフォーム」の法的位置づけ（資金決済法・第二種金融業等）も含め、必要に応じて専門家の確認が望ましい。 |

**検討の提案**: 受領者（オーナー）の本人確認・団体の実在性確認をどこまで行うか、法務・法域ごとに方針を検討する。

## 2. 弁護士に相談できる論点（相談内容が分かるように）

弁護士に送付する際に「何を相談したいか」が伝わるよう、論点を一覧化する。詳細な相談メモは **`docs/legal-consultation-brief.md`** の「5. 弁護士に相談したい論点」を参照。

| 論点 | 相談したい内容 |
|------|------------------|
| **利用規約・免責** | 仲介の範囲・保証しない旨・免責・責任制限・禁止行為・利用停止の根拠を記載した利用規約の文案・文言の確認。 |
| **免責条項の有効性** | 「寄付の判断は寄付者自身の責任」「プロジェクトの内容・真実性を保証しない」等が、消費者契約法・不法行為の観点でどこまで有効か。故意・重過失による免責の制限等の留意点。 |
| **クローン・悪用の免責** | About・README の「悪意あるクローン運営について公式は責任を負わない」旨の文言が、注意義務論等で考慮されうるか。修正・追加の推奨。 |
| **詐欺幇助・仲介の範囲** | 寄付金を経由せず決済導線のみ提供する場合の詐欺幇助・仲介責任の論点。事前審査なし・事後対応の方針について、取るべき記載・運用上の注意点。 |
| **税額控除・表明** | 「税額控除の可否は各自で確認」「表明の真正性を保証しない」旨の表示・規約について、不足論点や推奨文言。 |
| **利用停止・手続き** | 禁止行為の例・利用停止の手続きを規約に記載する場合、恣意性・不作為を争われないために必要な記載の程度。 |
| **本人確認・マネロン** | オーナーの本人確認・KYC を行わない方針について、規模・法域に応じて検討すべき法的要件。 |
| **第三者情報開示** | 開示の要否・範囲、事前に用意すべきデータの範囲、開示に際しての留意点（弁護士関与のタイミング、記録の残し方等）。 |

## 3. 推奨アクション（要約）

| 優先度 | 内容 |
|--------|------|
| 高 | **利用規約の整備**: 仲介の範囲、プロジェクト・オーナー表明の保証をしない旨、免責・責任制限、禁止行為・利用停止の根拠を記載。**弁護士確認**を推奨。 |
| 高 | **About・README**: 悪意あるクローン運営について責任を負わない旨の文言を**弁護士確認**のうえ記載。 |
| 中 | **税額控除まわり**: 「プラットフォームは表明の真正性を保証しない」「寄付者・オーナーが各自で確認すること」を表示・規約に明記。 |
| 中 | **プロジェクト掲載**: 禁止事項・ガイドラインを定め、違反時の凍結・削除と併せて運用方針として文書化。 |
| 低 | 本人確認・マネロン対策は、サービス規模・法域に応じて専門家に相談。 |

## 4. 第三者情報開示請求等への対応

裁判所の命令・法執行機関の照会・訴訟等で**第三者への情報開示**を求められた場合に、運営者が**何を提示すべきか**を整理する。開示の要否・範囲は弁護士判断を仰ぐが、**事前に取りまとめ可能な情報**を定義し、**管理画面から UI で出力できる**ようにする。

### 4.1 開示対象として整理すべき情報の種類

| 種類 | 内容 | 備考 |
|------|------|------|
| **ユーザー（アカウント）** | ユーザーID、メールアドレス、表示名、登録日時、更新日時、アカウント状態（active/suspended）、ロール、認証プロバイダ（Google/GitHub/Apple/Email の紐づけ有無）。パスワード・トークンは含めない。 | 開示請求の対象が特定ユーザーである場合 |
| **当該ユーザーがオーナーのプロジェクト一覧** | プロジェクトID、名前、説明概要、ステータス、登録日時、目標額・費用設定の要約。 | 同上 |
| **当該ユーザーの寄付履歴（寄付者として）** | 寄付ID、プロジェクトID・プロジェクト名、金額、日時、単発/定期、定期の場合はステータス（active/paused/cancelled）。寄付者識別子（donor_type, donor_id）は匿名化方針に合わせて必要時のみ。 | 同上 |
| **当該ユーザーに紐づく定期寄付（サブスク）** | 上記に含む。Stripe サブスクID があれば含める（入金履歴は Stripe 側）。 | 同上 |
| **特定プロジェクトの情報** | プロジェクトID、名前、説明、オーナー情報（ID・表示名・連絡用メール）、ステータス、登録日時、費用・目標設定、当該プロジェクトへの寄付一覧（金額・日時・donor は匿名化済みか識別子のみ、要領に応じて）。 | 開示請求の対象が特定プロジェクトである場合 |
| **運営操作の記録（あれば）** | 利用停止・停止解除・凍結・削除の実施日時と実施者（ホストまたはオーナー）。ログがある場合のみ。 | 判断の経緯を示すため |

上記は**開示の要否が認められた場合に提示しうる内容**のリストであり、請求があれば自動開示するものではない。開示の可否・範囲は**弁護士等の助言に従う**。

### 4.2 管理画面からの出力仕様

- **ホスト（運営者）**が、管理画面の UI から**開示用データをエクスポート**できるようにする。
- 対象の指定方法: **ユーザーID** または **プロジェクトID** を指定し、「開示用データを出力」を実行する。
- 出力形式: **JSON**（構造化データ。機械可読）。必要に応じて **PDF**（人間が読みやすい要約）を追加可能とする。
- 出力内容: 上記 4.1 の種類に従い、指定ユーザーまたは指定プロジェクトに紐づく情報を一括したパッケージとする。出力日時・プラットフォーム名をメタデータとして含める。
- アクセス制御: **ホスト権限を持つアカウントのみ**が実行可能とする。操作ログ（誰がいつどの対象で出力したか）を残すことを推奨する（実装は別フェーズ可）。

詳細な API・画面仕様は `docs/user-management-mock-plan.md` の「開示用データの出力」を参照。
